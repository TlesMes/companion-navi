> 작성: 2026-06-20. 범위: `architecture.md`가 **정적 계약**(모듈이 무엇을 주고받나)을 고정했다면,
> 이 문서는 그 위에서 도는 **동적 거동**(턴마다 텍스트 외에 무엇을 더 받고, 데몬이 언제 먼저 말하고,
> 관계·기억이 언제 어디서 바뀌나)을 모은다. `architecture.md`와 충돌하면 그쪽이 상위 — 여기선
> 비어있던 부분을 채우고 D번호로 미결을 추적한다.
>
> **상태 요약:** 능동성·상태 진화는 architecture.md에 설계돼 있으나 **미구현**.
> 턴 메타데이터 계약은 **설계 자체가 비어있음**(현재 Brain은 `full_text`만 반환). 이 셋이
> Phase 1(텍스트 뼈대)·Phase 2(음성화)를 넘어 "살아있는 반려"로 가는 핵심 미구현 축이다.

## 0. 핵심 통찰 — 빠른 경로와 느린 경로를 나눈다

이 문서 전체를 관통하는 원리. **첫 오디오 ~1초**(원칙 5)와 **LLM이 메타데이터의 최적 출처**라는
두 요구는 충돌한다. JSON 한 덩어리로 감정·사실·친밀도 신호를 다 받으면 스트리밍이 깨진다.

해법은 한 번의 LLM 호출에서 나오는 산출물을 **두 경로로 쪼개는** 것:

| 경로 | 무엇 | 언제 | 지연 제약 |
| :-- | :-- | :-- | :-- |
| **빠른 경로** | 발화 텍스트 토큰 (TTS로 직행) | 생성 즉시 스트리밍 | 첫 오디오 ~1초 — **블로킹 금지** |
| **느린 경로** | 사실 추출·친밀도 신호·요약 등 | 턴 종료 후, 비동기 | 사용자 체감 밖 — 별도(저가) 호출 허용 |

예외가 하나 있다: **이번 발화의 무드**는 합성 *전에* 필요해 느린 경로에 못 둔다 →
빠른 경로 맨 앞에 **선행 태그**로 끼운다(2.2). 이 "빠름/느림 분리"가 아래 모든 설계의 토대다.

## 1. 턴 메타데이터 계약 (D14 — 제안, 미결)

> **문제:** 현재 `BrainResult`(models.py)는 `full_text` + `usage`뿐이다. 텍스트 외에 데몬이
> 살아있으려면 받아야 할 부수 데이터가 더 있는데, 그 계약이 없다. 무엇을 / 어떻게 받을지 결정 필요.

### 1.1 받을 후보 데이터와 경로 배정

| 데이터 | 용도 | 경로 | 출처 후보 |
| :-- | :-- | :-- | :-- |
| **mood** (차분/신남/위로/평상) | 레퍼런스 클립 선택 → 톤 제어(D3) | **빠름**(선행 태그) | LLM 의도 or 결정론 감정분석 |
| **fact 후보** (key/value) | 장기기억 적재 (예: 반려견_이름=콩이) | 느림 | 턴 후 추출 호출 |
| **친밀도 신호** (관찰사실) | D9 산식 입력 (예: "사용자가 취약한 이야기 공유") | 느림 | LLM이 *관찰*만, 점수 계산은 데몬 |
| **응답 기대 여부** | 능동 발화가 답을 요구하나 / "대답 안 해도 OK"인가 | 빠름(능동만) | 능동성 엔진이 부여 |
| **주제 종료/전환 신호** | 세션 경계·요약 트리거 | 느림 | 턴 후 분석 |
| **언어** | 응답 언어·음성 레퍼런스 정합 | 빠름 | STT lang or LLM |

### 1.2 mood를 스트리밍 안 깨고 받는 법 — 선행 태그

LLM이 응답 **첫 토큰**으로 무드를 뱉고, 데몬이 그걸 읽어 레퍼런스를 고른 뒤 **나머지 본문만** TTS로
흘린다. 무드 태그 자체는 절대 합성하지 않는다(읽히면 "대괄호 기쁨" 사고 — 현재 페르소나에 감정
태그를 넣으면 안 되는 이유와 동일).

```
LLM 출력 스트림:  [mood:comfort] 고생했네. 뭐가 제일 힘들었어?
                  └─ 데몬이 흡수      └─────── TTS로 직행 ───────┘
                     → ref_clip = comfort_pool 선택
```

- **장점:** LLM의 의도 기반 무드(가장 정확) + 스트리밍 유지. 추가 호출 0.
- **제약:** 출력 포맷을 강제해야 함(첫 토큰이 반드시 태그). 페르소나 few-shot이 아니라 **시스템
  프롬프트 말미의 출력 규칙**으로 강제하고, 파서는 태그 누락 시 `mood:neutral` 폴백.
- **대안(결정론):** trigger·최근맥락에 규칙/소형분류기로 무드 결정 → 원칙 3에 더 충실하나 미묘함↓.
  **권장: mood는 선행 태그(LLM), 단 폴백은 결정론.**

### 1.3 느린 경로 = "성찰(reflection) 패스"

턴이 끝나면(사용자 체감 밖에서) 별도의 **저가 LLM 호출**이 직전 턴을 분석해 fact·친밀도 신호·
요약을 뽑는다. `architecture.md` 4.5의 `extract_and_store_facts`가 이 자리다 — 단 현재 **호출처가
없다**(아무도 안 부름). 배치(턴마다 vs N턴마다 vs 유휴 시)는 원가·정확도 트레이드오프로 D14에 포함.

> **안전 불변식:** 느린 경로가 만든 어떤 신호도 **모드 게이트(수면/DND)를 건드리면 안 된다**(원칙 3).
> LLM은 "관찰"만 보고하고, "잘까 말까/말 걸까 말까"는 항상 결정론 규칙이 정한다.

### 1.4 환각 방어 — LLM은 신호, 계산은 데몬

친밀도를 LLM에게 "+5 줘"라고 시키면 환각·아부로 부풀려진다. 그래서 LLM은 **관찰 가능한 사실**만
보고하고(예: `shared_vulnerability: true`), 점수로의 변환은 **데몬의 결정론 산식(D9)**이 한다.
같은 신호라도 `persona.dynamics` 계수에 따라 성격마다 다르게 반영된다(6장 데이터모델 `dynamics`).

## 2. 능동성 — 먼저 말 걸기 / 맥락 창조

> **상태:** `architecture.md` 4.4(Heartbeat 3층)·4.11(루프)·5(모드 상태머신)에 설계 완료, **미구현.**
> 이 프로젝트의 정체성("두뇌는 빌리고 살아있음은 데몬이 만든다")이 가장 직접적으로 구현되는 곳.

### 2.1 3층 구조 (요약 — 상세는 architecture.md 4.4)

```
1층 모드 게이트 (결정론, 모델 금지):  지금 말해도 되나? SLEEP/DND/SNOOZE면 차단
2층 타이밍 (가중치+jitter):           지금이 말 걸 타이밍인가? (마지막 상호작용 후 경과·시간대)
3층 주제 도출 (소형 LLM 허용):        건다면 무슨 주제로? (기억·날씨·시간대·피드)
```

핵심: **"언제(1·2층)"는 멍청하지만 확실한 규칙, "무엇(3층)"만 모델.** 안전이 걸린 1층을 LLM에
맡기지 않는 게 절대 규칙.

### 2.2 "인간처럼 맥락을 창조" = 시간 인지 + 기억 인출

먼저 거는 말이 자연스러우려면 데몬이 **시간의 흐름을 느껴야** 한다. 핵심 입력은
`last_interaction_at`(마지막 상호작용 시각)과 현재 시각의 차. 여기서 맥락이 만들어진다:

- 오래 침묵 → "오랜만이네, 바빴어?" (경과 시간 인지)
- 아침 첫 활성 → "잘 잤어?" (시간대 + 기상 스케줄)
- 어제 들은 일 후속 → "어제 그 발표 어떻게 됐어?" (fact + 시간 경과)

즉 맥락 창조는 마법이 아니라 **(시간 신호 × 기억 인출)을 3층 주제 도출에 먹이는 것**. 기억이 쌓일수록
후속 질문이 풍부해진다 → 3장(상태 진화)과 직결.

### 2.3 안전·피로 가드 (architecture.md 7장 설정값)

- `proactive_daily_cap`: 하루 능동 발화 상한 (원가·피로 방지)
- `min_gap_between_initiations`: 능동 발화 최소 간격
- 능동 발화 후 **무응답이어도 재촉 금지** — "대답 안 해도 괜찮은" 경험 (루프 4.11 주석)
- 피드 주제는 **수집 단계에서 자극적·부정적 이슈 규칙 필터**(아침 첫마디 안전, 4.10)

### 2.4 테스트 가능성 (잊기 쉬움)

능동성은 시간 의존이라 **시계를 주입 가능하게** 설계해야 단위 테스트가 된다(`now`를 인자로 받는
현 계약이 이미 그 형태 — 유지). 실제 타이밍 품질은 종이로 결정 불가, `interaction_log`로 데이터
수집 후 튜닝(D12와 동류).

## 3. 상태 진화 — 관계·기억은 어디서 계산되고 어디서 바뀌나

> **질문 직답:** "친밀도/기억이 시간·축적에 따라 변하는 계산은 어디서?" → **계약상 메모리 모듈(4.5),
> 트리거는 턴 종료 시점. 단 현재는 산식도 호출처도 없는 빈 골격.**

### 3.1 현재 구현 실태 (정직하게)

| 요소 | 계약 | 현재 구현 (store.py) | 갭 |
| :-- | :-- | :-- | :-- |
| 친밀도 저장 | `get/update_intimacy` | ✅ 단순 가감만 | **부르는 곳이 없음** |
| 친밀도 산식 | D9 (감쇠·히스테리시스) | ❌ delta 그대로 더함 | Phase 4 |
| 사실 추출 | `extract_and_store_facts` | ❌ 미구현 | 느린 경로 미구축 |
| 사실 인출 | `recall_relevant_facts` | ❌ 미구현 | 임베딩(D6) 미정 |
| 단기기억 | `recall_recent_for_user` | ✅ 동작 | — |

즉 지금 데몬은 "어제 대화는 기억"(단기)하지만 **관계가 깊어지지도, 사실을 구조화하지도 않는다.**

### 3.2 변경이 일어나야 할 트리거 지점

```
[턴 종료]
   ├─(빠름) 단기기억 append_turn          ← 이미 동작
   └─(느림, 성찰 패스) ────────────────────────────────────
        ├─ extract_and_store_facts        → fact 테이블 (신규/갱신/모순해결)
        ├─ 친밀도 신호 수집 → D9 산식 → update_intimacy
        └─ 필요 시 요약 → memory_embedding (의미 인출용)

[페르소나 전환]  intimacy가 임계값 교차 시 → Conductor가 다음 턴부터 다른 profile 선택
                  (card.py profile_for 이미 단계 선택 지원 — 데이터만 채우면 됨)
```

- **계산 위치:** D9 산식은 메모리 모듈 안(결정론). LLM은 신호만 제공(1.4).
- **변경 위치:** `update_intimacy` / fact 테이블 — 둘 다 메모리 모듈. **느린 경로에서만** 호출.

### 3.3 단조증가 방지 (D9 — 잊으면 망가짐)

친밀도가 대화량에 비례해 무한 상승하면 페르소나가 금방 최종단계로 점프해 서사가 붕괴한다. D9 산식
필수 요건:

- **감쇠(decay):** 오래 상호작용 없으면 친밀도 하락 — "안 보면 멀어진다"
- **히스테리시스:** 페르소나 전환 경계에서 진동 방지 (올라갈 땐 80, 내려갈 땐 70 식 이중 임계)
- **성격별 계수:** `persona.dynamics`(intimacy_gain_rate 등)를 산식에 곱함 — 무뚝뚝한 캐릭터는 천천히

### 3.4 기억의 모순·망각 (잊기 쉬움)

축적만 설계하고 빠뜨리기 쉬운 것들:

- **사실 모순 해결:** "반려견 이름 콩이" → 나중에 "두부". 덮어쓸까/이력 남길까? (fact `updated_at`·
  `confidence`로 최신 우선 + source_turn_id 추적)
- **망각 정책:** 단기 턴 무한 누적 → 인출 비용·노이즈↑. 오래된 턴은 요약 후 폐기(단기→장기 승격).
- **프라이버시:** fact는 민감정보. 로컬 전용 저장이 곧 제품의 해자(외부 유출 0). 삭제 요청 경로 필요.

## 4. 잊기 쉬운 횡단 고려사항

1. **닭-달걀 무드:** 이번 발화 무드는 본문 *전*에 필요 → 선행 태그(2.2)로만 해결. 절대 후행 JSON 불가.
2. **메타데이터는 안전 게이트를 못 건드린다:** 수면/DND 판단은 영원히 결정론(원칙 3). LLM 신호는 자문용.
3. **출력 포맷 강제 = 페르소나 오염 위험:** mood 선행 태그를 few-shot에 넣으면 모델이 본문에도 흉내냄.
   시스템 프롬프트의 *출력 규칙* 섹션으로 분리하고, 페르소나(말투)와 격리.
4. **원가:** 느린 경로(성찰 패스)는 턴마다 호출하면 LLM 비용 2배. N턴 묶음·유휴 시 배치로 상각.
5. **다국어:** 응답 언어·음성 레퍼런스 정합(일/한). STT lang ↔ 페르소나 출력 언어 ↔ ref_lang 일치 보장.
6. **관찰가능성:** mood 오선택·친밀도 급변을 사후 검증하려면 `interaction_log`/`usage_log`에 무드·신호도
   남겨야 튜닝 가능.

# [테스트 계획] Zero-shot TTS 전환 검증 (D3)

> [TTS_전환.md](./TTS_전환.md)에서 결정한 **Supertonic → zero-shot voice cloning** 피벗을 코드로 옮기기 전, 가정을 단계별 관문으로 검증하는 계획. 각 단계는 게이트 — 통과해야 다음으로 간다.
> 호스트: AMD RX 6600 XT(RDNA2, VRAM 8GB) / ROCm. 한국어 퍼스트.
> 관련 D번호: **D3(TTS 음색)**. 결과는 04_progress.md와 커밋 본문에 D3 근거로 적재.

---

## 0. 검증할 핵심 가정 (왜 이 순서인가)

피벗 문서가 깔고 있는 미검증 가정 셋. **싸고 잘 깨지는 것부터** 친다 — 음색이 아무리 좋아도 ROCm에서 안 돌면 무의미하므로 환경이 0순위.

| # | 가정 | 깨질 확률 | 깨지면 |
| :-- | :-- | :-- | :-- |
| A | RX 6600 XT(gfx1032)에서 ROCm + PyTorch가 **안정적으로** 돈다 | **높음** (비공식 지원) | 프로젝트 방향 재검토 (CPU 폴백/클라우드) |
| B | zero-shot 모델이 **한국어 + 캐릭터 음색**을 만족스럽게 복제한다 | 중간 | 모델 교체 또는 fine-tuning |
| C | 그 음색을 **TTFA ~1초**로 합성한다 (이 GPU에서) | 중간~높음 | 더 빠른 모델 우선 / 0.5초 목표 하향 |

> **속도 목표 정정:** 피벗 문서의 "TTFA 0.5초"는 NVIDIA 기준 수치(4060Ti RTF 0.028 등)에서 나온 낙관치. 6600 XT + ROCm에서는 **1차 목표 TTFA ~1초**(CLAUDE.md "첫 오디오 ~1초"와 정렬), 0.5초는 최적화 후 도전 목표로 둔다. Supertonic 실측 0.6초(웜, CPU)가 베이스라인 — 신규 모델이 이보다 나쁘면 전환 명분이 약해진다.

## 1. 단계별 게이트

### Stage 0 — ROCm 환경 성립 (가정 A) · 반나절

모델 얘기 전에 이것부터. 가장 치명적이고 가장 잘 깨지는 가정.

- **셋업:** Ubuntu + ROCm + `HSA_OVERRIDE_GFX_VERSION=10.3.0`(gfx1032→gfx1030 위장)으로 PyTorch가 카드를 인식하는지.
- **측정:**
  1. `torch.cuda.is_available()` → `True`, `torch.cuda.get_device_name()`이 6600 XT를 가리킴
  2. 중간 크기 텐서 matmul을 GPU에서 5분 연속 반복 → 크래시·행(hang)·VRAM 누수 없음
  3. VRAM 여유 확인 (LLM은 API라 6GB 마진 가정이 맞는지)
- **합격 기준:** 5분 연속 연산 무크래시 + GPU 활용 확인. **불안정하면 여기서 멈추고** 환경을 잡거나(ROCm 버전·커널), CPU 폴백/클라우드(원 관문)로 방향 전환 판단.
- **산출 코드:** `scripts/bench_rocm.py` — 환경 진단 + 스트레스 루프 + VRAM 로깅.

### Stage 1 — 음색 품질 청취 비교 (가정 B, D3 본체) · 1~2일

속도 무시. **품질만** 본다. 배치로 미리 뽑아 귀로 블라인드 비교.

- **후보:** GPT-SoVITS V2 / F5-TTS / CosyVoice2 (피벗 문서 3종)
- **입력 조건 (중요):**
  - **실제 캐릭터 오디오 샘플** + **한국어 대사**로 합성 (영어 데모만 보고 결정 금지)
  - 교차언어가 목표면 **일본어 레퍼런스 → 한국어 발화** 케이스도 포함
  - 3~10초 정제 레퍼런스(BGM·효과음 제거) — 전처리는 Stage 1.5 참조
- **평가 항목 (문장 5~10개, 블라인드 점수 1~5):**
  - 음색 닮음도 / 한국어 발음·억양 자연스러움 / 잡음·아티팩트 / 감정선 표현
  - ※ GPT-SoVITS는 한·일 출력에 잡음 아티팩트 보고 있음 → 이 항목 집중 확인
- **합격 기준:** 1개 이상 모델이 **Supertonic F1 대비 음색 만족도 우위** + 한국어 치명적 결함 없음. 전부 미달이면 fine-tuning(1분 데이터) 시도 또는 원 관문(클라우드) 복귀.
- **산출 코드:** `scripts/try_clone.py` — 모델·레퍼런스·문장 받아 샘플 wav 일괄 생성(어댑터별).

### Stage 1.5 — 레퍼런스 전처리 파이프라인 · Stage 1과 병행

캐릭터 대사 파일 → 깨끗한 3~10초 보컬 레퍼런스 추출 (피벗 행동지침 2번).

- BGM·효과음 제거(Vocal Remover 계열) → 무음 트림 → 길이·샘플레이트 정규화
- **합격 기준:** 사람이 듣기에 보컬만 깨끗이 남고, Stage 1 합성 품질이 raw 대비 향상.
- **산출 코드:** `scripts/prep_reference.py`.

### Stage 2 — 추론 속도 실측 (가정 C) · 1일

**진짜 6600 XT 검증.** Stage 1 음색 합격 모델만 측정.

- **지표:**
  - **TTFA** (첫 음성까지) — 문장 길이별(짧음/중간/긺) 각각
  - **RTF** (오디오 길이 대비 합성 시간)
  - **VRAM 점유** (8GB 안에 드는지)
  - **콜드 vs 웜** (첫 발화 워밍업 비용 — Supertonic은 콜드 1.4s/웜 0.6s였음)
- **방법:** 워밍업 후 10회 반복, 평균 + 최악값(p95) 둘 다 기록.
- **합격 기준:** 웜 **TTFA ~1초 내외**. 1.5초 초과면 빨간불 → 음색 최상이라도 F5-TTS(플로우매칭, 추론 빠름) 등으로 우선순위 조정. VRAM 8GB 초과면 탈락.
- **산출 코드:** `scripts/bench_tts_latency.py` — Stage 1 어댑터 재사용, 타이밍·VRAM 계측.

### Stage 3 — 스트리밍 통합 검증 · 음색·속도 통과 후

실제 파이프라인 조립 후 **체감 지연**과 바지인 확인.

- LLM 텍스트 스트리밍 → 문장 분할 → TTS 큐 → 재생까지 연결. (기존 `SupertonicMouth`의 문장청크 스트리밍 구조 재사용 — 계약 4.8 그대로)
- 문장 단위 버퍼링이 실제로 끊김 없이 이어지는지 (합성 N+1이 재생 N과 겹치는지)
- **바지인/킬스위치:** 끼어들기 감지 후 ① 재생 중단 ② TTS 큐 clear ③ LLM 스트림 종료 ④ STT 전환 — 각 단계 실제 지연(ms) 측정
- AEC: 자기 출력 재유입 오작동 방지 (이어셋 전제 + WebRTC AEC 여지)
- **합격 기준:** 자연스러운 연속 발화 + 끼어들기 체감 즉각(중단 ~0.2초 내, Supertonic 실측 수준).

## 2. 게이트 흐름 요약

```
Stage 0 (ROCm 성립?) ──실패──> 방향 재검토 (CPU/클라우드)
   │ 통과
Stage 1 (음색 만족?) ──실패──> fine-tuning / 원 관문 복귀
   │ 통과 (+ Stage 1.5 전처리)
Stage 2 (속도 ~1초?) ──실패──> 더 빠른 모델로 우선순위 조정
   │ 통과
Stage 3 (스트리밍·바지인 체감 OK?) ──> D3 확정 → Mouth 실어댑터 구현
```

원칙 둘: **(1) 각 단계는 통과해야 다음으로.** **(2) 모든 결과는 숫자로 기록**(TTFA·RTF·VRAM·바지인 지연 ms·음색 점수) — "괜찮은 것 같다" 금지, D3 결정 근거로 남긴다.

## 3. 코드 착수 매핑

테스트 단계가 그대로 스크립트가 되고, D3 확정 후 어댑터가 된다. 기존 `scripts/try_*.py` 컨벤션과 Mouth 계약(01 문서 4.8) 재사용.

| 단계 | 산출 코드 | 비고 |
| :-- | :-- | :-- |
| Stage 0 | `scripts/bench_rocm.py` | 환경 진단·스트레스·VRAM |
| Stage 1.5 | `scripts/prep_reference.py` | 레퍼런스 전처리 |
| Stage 1 | `scripts/try_clone.py` | 모델별 음색 샘플 생성 |
| Stage 2 | `scripts/bench_tts_latency.py` | TTFA·RTF·VRAM 계측 |
| Stage 3 | 기존 파이프라인 + 바지인 계측 | `SupertonicMouth` 스트리밍 구조 재사용 |
| 확정 후 | `navi/mouth/<model>.py` | Mouth 계약 3메서드 구현, `create_mouth()` 등록, `_PENDING_D3` 해제 |

**어댑터 경계 유지:** 신규 TTS도 반드시 Mouth 계약 뒤에 둔다(벤더 비종속 원칙). Supertonic 어댑터와 동일 인터페이스라 교체는 어댑터 한 장.

**의존성:** 음성 의존성은 기존대로 `[voice]` extra로 분리해 텍스트 뼈대는 가볍게 유지. 신규 모델·ROCm PyTorch는 별도 extra 검토.

## 4. 선행 결정/확인 사항

- ROCm 설치는 Ubuntu 기준 — **WSL2로 진행** (2026.06.15 결정). 데몬 본체는 Windows에서 계속 돌고, ROCm·GPU 가속 연산만 WSL2 Ubuntu에서 처리하는 형태. Stage 0에서 WSL2 + ROCm + gfx1032 조합 안정성도 같이 확인한다.
- 캐릭터 오디오 샘플의 저작권·사용 범위 (개인 프로젝트 전제)
- LLM 스트림 강제 종료 비용: 매 끼어들기마다 재연결되므로, API별 abort/취소 토큰 지원 여부를 Brain 어댑터 계약에 반영

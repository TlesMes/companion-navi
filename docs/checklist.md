## E. 데몬 기동 정상화 — 진행 중 (2026.07.19 시작, A·B보다 우선)

`--voice` 부팅이 어떤 카드로도 죽던 버그에서 출발한 묶음. 순서 강제: E1 → E2 → E3 → E4.

      config 기본 supertonic + 카드 gptsovits kwarg 무조건 주입 → `SupertonicMouth(gpt_ckpt=…)`
      TypeError. 기본 카드 navi.yaml도 해당돼 **보편적 실패**였다. `--mouth` 불요.
      `os._exit(0)`의 traceback 은폐도 함께 수정(실패 시 exit 1). 241 tests green.
- [ ] **E2. 표준 실행 스크립트 `scripts/run_navi.ps1`** — venv 선택·필수 인자·cwd 고정을 한 곳에.
      무인자 기본 = **음성 + 웨이크워드 + Claude brain**(`--brain anthropic`으로 넘김, D1 보류라
      config는 안 건드림). GUI 대기 화면 안내 문구 정정(현재 **틀린 명령**을 가르침)도 이 PR.
      계획: `~/.claude/plans/cozy-tinkering-walrus.md`.
- [ ] **E3. 페르소나 음색 호환 게이팅(GUI 비활성화)** — E2 직후. 엔진은 부팅 시 고정인데
      **어떤 카드로 부팅했느냐**가 다른 페르소나의 목소리 사용 가능 여부를 좌우한다(기본값으로
      부팅하면 다른 4장 중 3장이 반쪽 전환). 안내 로그보다 **버튼 비활성화 + 사유 툴팁**이
      직관적이라 판단(2026.07.19). 인격만 바뀌고 목소리는 그대로인 반쪽 정체성은 연속성 원칙에
      어긋나므로 전면 비활성화.
      **비활성화 조건은 벤더 불일치뿐** — 같은 벤더의 ckpt 교체(핫스왑 0.96~1.65s, A1에서 검증)는
      **정상 기능이라 대상 아님**. 표로 고정:
      | 상황 | 비활성화 |
      | :-- | :-- |
      | 벤더 불일치(supertonic 세션 ← gptsovits 카드) | **O** |
      | 같은 벤더·ckpt 다름 (aris_base↔aris) | X — 핫스왑 정상 |
      | 같은 벤더·ckpt 같음 (aris_base↔example_jp) | X — 레퍼런스 교체 |
      | voice 섹션 없음 (example_kr) | X — 목소리 유지가 의도 |
      | fine-tune ckpt 파일 부재 | **O** — 단 E4 선행 필요 |
      **함정 2개:** ① 텍스트 모드는 `pipeline=None`이라 순진하게 짜면 **전 카드가 비활성화**된다 →
      조건에 "음성 세션" 포함 필수 ② 번들 없는 카드는 대상 아님.
      구현: `SwapRuntime.list_personas()`가 이미 카드를 전수 로드하므로 `/personas` 응답에 필드
      하나 추가 + GUI가 읽기. `feat(control)` 별도 PR.
- [ ] **E4. fine-tune ckpt 존재 검사** — E3 게이팅의 **데이터 소스**. 지금은 카드가 가리키는 ckpt가
      없으면 `_load_weights`의 torch가 날것의 에러를 낸다 → `_load_weights` 앞에 `os.path.isfile`
      한 줄 + "카드 aris.yaml이 가리키는 ckpt가 없습니다: …" 수준의 안내로.
      **`_resolve_base_ckpts`와는 층위가 다르다**(2026.07.19 정리): 그쪽은 파일 부재가 아니라
      **weight.json 폴백**(경로 미지정 시 엔진이 "지난번 가중치"를 집어듦 — 아리스로 돌린 뒤 base
      카드로 부팅하면 아리스 목소리)을 막는 결정론 장치라 **게이팅이 생겨도 없앨 수 없다**.
      또 base 가중치는 **엔진 공용 자산**이라 페르소나 단위 GUI 게이팅이 못 덮는다(없으면 zero-shot
      카드가 전부 회색 — 원인은 페르소나가 아니라 환경). 판정은 데몬, GUI는 관찰·제어라는 원칙에도
      부합. **정리: base=엔진 공용→데몬 부팅 검사(현행 유지) / fine-tune=페르소나별→명확한 에러 +
      GUI 게이팅.**
      실사용 근거: `voice_ref/`와 `personas/aris*.yaml`이 gitignore(로컬 전용)라 **다른 머신·새
      클론에서 즉시 발생**하는 시나리오다.

## B. Phase 3 본류 — 순서 4 튜닝 → 순서 5

- [ ] **B1. 조건 산식 재구상(변동확률)** — `should_initiate`를 고정 간격 bool에서 **tick 기반 hazard**

      ("선제 발화 응답률 > 무시율")은 로그 없이 닫을 수 없다. **B1 먼저, 그 위에서 튜닝**(진행 원칙 2).
